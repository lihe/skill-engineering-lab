AI Agent Skill 工程化实践

编写、评测、治理与持续迭代的一体化方法论

STATUS · PORTFOLIO EDITION DATE · 2026-06-09 PROJECT · Skill Engineering Lab
TL;DR
Skill 不是长 Prompt，而是一个由路由入口、任务契约、参考资料、确定性脚本、评测资产和治理资产组成的能力包。成熟的 Skill 不应只证明“能跑”，而应证明它在正确场景稳定触发、相对无 Skill 基线带来可验证净增益，并能通过失败样例、回归集和版本记录持续治理。

01三个核心判断

第一，Skill 不是把 Prompt 写长。它的价值在于让 Agent 在正确场景加载正确知识，沿着稳定执行路径完成任务，并避免把所有上下文一次性塞进主提示。

第二，评测不是证明它能跑。真正要测的是该触发时会不会触发、不该触发时会不会克制、触发后是否带来净增益，以及这种增益是否值得额外成本。

第三，治理不是可选项。没有治理，Skill 会逐渐变成上下文税、规则堆和无法退休的历史包袱。

CORE CLAIM
成熟的 Skill，不是某次任务里“刚好成功”的说明书，而是能稳定触发、持续带来可验证净增益、并能被团队长期评测和治理的能力资产。

02重新定义 Skill

Skill = 路由入口
      + 任务契约
      + 按需加载的知识与模板
      + 确定性脚本
      + 评测资产
      + 治理资产
组成部分主要职责工程含义
name / description让主 Agent 发现并决定是否使用它是路由入口，不是装饰摘要。
SKILL.md定义任务契约、主流程、边界与关键规则首要读者是 Agent，而不是人类。
references / templates承载长知识、模板、示例和分支说明用于渐进披露，避免主文档膨胀。
scripts承载校验、解析、格式转换等确定性动作能脚本化的事情，不要靠自然语言反复提醒。
evals / governance验证净增益，沉淀回归和版本决策决定 Skill 是一次性技巧还是团队资产。

04高质量 Skill 的编写原则

ROUTE
先定义边界
明确解决什么任务、典型输入、成功标准、不负责什么，以及最容易和哪些 Skill 冲突。
CONTRACT
主文档短硬
SKILL.md 只放任务契约、主流程、关键边界、必检项和何时读取哪个 reference。
SCRIPT
确定性逻辑脚本化
JSON 解析、schema 校验、文件结构检查、格式转换、固定规则判定都应优先交给脚本。
COMMON FAILURE
很多 Skill 失败不在正文，而在 description。写得太泛会过度触发，写得太窄会漏掉真实表达；只写能力、不写场景，会让主 Agent 无法判断何时使用。

05评测要看 lift，不只看 pass

每条关键样例至少跑三类配置：without_skill、with_skill、old_skill。真正值得看的不是绝对分，而是差值。

lift = with_skill pass rate - without_skill pass rate
net_value = quality lift - token/time/complexity cost
层级核心问题如果出问题，优先改哪里
触发层该触发时是否触发，不该触发时是否沉默description、负面样例、边界
结果层最终产物是否可用主流程、模板、schema、示例
执行层是否遵守规则与约束SKILL.md 指令、checklist
效率层是否更省、更稳、更少绕路渐进披露、脚本化、流程收缩
泛化层换说法、换场景、换噪声时是否仍成立补边界、补反例、扩展样例分布
治理层是否便于回归、定位、维护和退休版本记录、失败归因、回归集

06从失败样例到迭代

badcase 不是问题清单，而是优化驱动器。每个失败都应该被翻译成可执行动作：改入口、改契约、补示例、拆 references，或者把确定性逻辑下沉到 scripts。

{
  "case_id": "boundary-sparse-brief",
  "run_config": "with_skill_v1",
  "human_expected": "稀疏输入也应触发视频创作 Skill",
  "observed_behavior": "v1 没有触发",
  "failure_pattern": "触发不匹配",
  "root_cause": "description 没覆盖创作者式稀疏表达",
  "suggested_action": "扩展 description 并加入回归集"
}
