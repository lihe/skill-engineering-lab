AI Agent Skill 工程化实践：编写、评测与治理

一套把 Skill 从个人技巧升级为团队能力系统的方法论

STATUS · PORTFOLIO EDITION DATE · 2026-06-09 SCOPE · WRITING / EVAL / GOVERNANCE / HARNESS PROJECT · Skill Engineering Lab
Operating Model 6 Assets 路由、契约、参考、脚本、评测、治理
Eval Stack 6 Layers 触发、结果、执行、效率、泛化、治理
Minimum Cases 12+ 核心、边界、扩展、负向与回归样例
Decision Ship / Split / Retire 用证据决定 Skill 生命周期
TL;DR
Skill 不是长 Prompt，而是由路由入口、任务契约、按需加载的知识、确定性脚本、评测资产和治理资产组成的能力包。成熟 Skill 的判断标准不是“某一次能跑”，而是它能否在正确场景稳定触发，相对无 Skill 基线带来可验证净增益，并通过 badcase、回归集、版本记录和外层 Harness 长期治理。

01三个核心判断

这不是一份“怎么把 Prompt 写长一点”的文档，而是一份把 Skill 当成工程资产来建设的方法论。它关注的是编写、评测、治理、持续迭代如何串成闭环。

Judgment A
Skill 不是长 Prompt
价值不在写得多，而在正确触发、稳定执行、控制上下文成本，并沉淀为可复用能力。
Judgment B
评测不是证明能跑
真正要测的是触发、克制、净增益、泛化稳定性，以及增益是否值得成本。
Judgment C
治理不是可选项
没有治理，Skill 会变成上下文税、抢触发规则堆，以及无法回归和退休的历史包袱。
Core Claim
成熟的 Skill，不是某次任务里“刚好成功”的说明书，而是能稳定触发、持续带来可验证净增益，并能被团队长期评测和治理的能力资产。

02重新定义 Skill：从 Prompt 到能力包

很多团队默认把 Skill 当成“保存起来的 Prompt”。这个理解太浅，也会直接误导设计方式。更准确的定义是一个可路由、可执行、可验证、可维护的能力包。

Skill = 路由入口
      + 任务契约
      + 按需加载的知识与模板
      + 确定性脚本
      + 评测资产
      + 治理资产
组成部分主要职责工程含义
name / description让主 Agent 发现并决定是否使用这是路由入口，不是装饰信息。入口写不好，正文再好也没有意义。
SKILL.md定义任务契约、主流程、边界与关键规则它更像执行说明书，首要读者是 Agent，而不是人类。
references / templates / examples承载长知识、模板、示例和分支说明用于渐进披露，避免主文档膨胀成上下文税。
scripts承载校验、解析、格式转换和生成等确定性动作凡是能脚本化的事情，尽量不要交给自然语言反复提醒。
evals验证 Skill 是否带来真实增益目标不是看起来不错，而是验证 with_skill 相比 without_skill 是否更好。
governance assets支持回归、版本记录、错因归档与退休决策决定 Skill 是一次性技巧，还是团队能力资产。
Route First
description 不是摘要，而是给主 Agent 看的 API docstring。一个好的 description 至少要回答：什么时候应该用、用来解决什么问题、什么时候不该用。

03生命周期视角：写完只是开始

与其把 Skill 看成“写完一个文件”，不如把它看成一条完整生命周期。真正的成熟发生在评测、失败归因和治理闭环里。

04编写原则：从 Prompt 思维切到能力包思维

最稳的路径不是先想一个“通用 Skill”，而是先手动完成 1 到 2 次真实任务，记录关键步骤、判断点与失败点，再抽象成 Skill。

  1. 先做真实任务。 真实任务会暴露输入缺失、边界含混、工具选择和输出验收问题，这些摩擦比抽象设想更有价值。
  2. 先定义能力边界。 在写正文前，先说清它解决什么、典型输入是什么、可接受输出是什么、明确不负责什么。
  3. 让主文档短、硬、可执行。 SKILL.md 只放任务契约、主流程、关键边界、必须遵守的规则和按需加载指令。
  4. 用渐进披露管理上下文。 领域知识、长示例、反例、模板和特殊分支流程应该拆到 references，而不是堆在主文件里。
  5. 能脚本化的，不要靠自然语言念咒。 结构化校验、格式转换、固定规则判定、低容错高频步骤都应优先脚本化。
交给 Skill 文本交给脚本
模糊判断、流程组织、分支选择、何时澄清用户、何时调用工具机械执行、结构化校验、可重复转换、固定规则判定、文件结构检查
Human In The Loop
成熟 Skill 不等于完全自动。需求歧义、多方案取舍、风格偏好、高风险操作、关键输入缺失时，好的 Skill 应该知道什么时候继续，什么时候停下来问人。

05常见误区：上下文税是怎样形成的

误区典型表现后果
把 Skill 写成长 Prompt背景很多，执行规则很少，主文件不断膨胀触发后负担重，关键约束反而被淹没
description 只写能力“用于文档处理”“帮助完成分析”等笼统描述不触发或乱触发
确定性逻辑交给模型猜格式校验、解析、工具参数都靠自然语言要求不稳定、难复现、难评测
不写负面边界没有“何时不要使用”相邻任务大量串扰
没有定义完成标准只写“帮我更好地做某事”后续 eval 与 badcase 都无从落地

06评测原则：看 lift，不只看 pass

评测一个 Skill，至少要回答 6 个问题：该触发时能否触发，不该触发时能否克制，触发后是否更好，提升是否稳定，换表达或噪声后是否仍成立，提升是否值得 Token、时延和维护成本。

lift = with_skill pass rate - without_skill pass rate
net_value = quality lift - token/time/complexity cost
层级核心问题典型证据优先修哪里
触发层该触发时是否触发，不该触发时是否沉默trace 是否加载 Skill，negative case 是否未加载description、边界、负面样例
结果层最终产物是否可用文件存在、测试通过、报告完整、接口正确主流程、模板、schema、示例
执行层是否遵守 Skill 规则与约束工具、SDK、命令、流程是否符合预期SKILL.md 指令、checklist、红线
效率层是否更省、更稳、更少绕路token、耗时、tool call 数、重试次数渐进披露、脚本化、流程收缩
泛化层换说法、换场景、换噪声时是否仍成立改写 case、真实历史 case、新批次数据补边界、补反例、扩展 case 分布
治理层是否便于回归、定位、维护与退休badcase 归档、版本记录、回归集、benchmark history治理资产建设与平台接入

07样例库与对照实验：不要只堆 prompt

单个 Skill 起步时，建议先做一个结构完整的小样例库，而不是盲目追求数量。最小可行样例库应覆盖核心、边界、扩展、负向和回归场景。

6-8 core cases 3-4 boundary cases 2-3 extension cases 2+ negative cases badcase reflux 
{
  "id": "case_id",
  "prompt": "用户输入",
  "should_trigger": true,
  "expected_checks": ["check_1", "check_2"],
  "case_type": "core | boundary | negative | regression",
  "risk_tag": ["trigger_boundary"],
  "source": "manual_discovery | online_case | badcase"
}

每条关键 case 至少跑三类配置：without_skill 看基础模型本来会不会，with_skill 看 Skill 是否带来增益，old_skill 看新版本是否优于旧版本。条件允许时再加入 ablation 与 shadow eval。

08评分与诊断：把 badcase 翻译成动作

评分建议使用 0 / 1 / 2 分制，并同步记录触发召回率、误触发率、结果通过率、指令违背率、trial 波动、token / time delta、新旧版本差值。不要只盯总平均数，很多严重问题都藏在平均数里。

Deterministic First
能用代码检验的，不要交给 LLM Judge。文件存在、JSON 合法、构建测试、工具参数、禁用项、trace 调用都应优先用确定性检查，LLM Judge 只补充语义质量。
根因类型典型表现真正的问题优先动作
路由问题该触发不触发，或相邻任务误触发入口和边界写得不对先改 description、补负面 case
规则问题触发后执行偏离，忘关键步骤任务契约不清、优先级不清改 SKILL.md、加 checklist、加示例
工程问题机械步骤重复出错，输出结构不稳本可脚本化的内容仍在靠模型猜补脚本、schema、模板、校验器
评测问题所有配置都失败，或 grader 明显不可信case 设计不准，评测本身有偏差先修 case / grader，再修 Skill
 
{
  "case_id": "boundary-sparse-brief",
  "run_config": "with_skill_v1",
  "human_expected": "稀疏输入也应触发视频创作 Skill",
  "observed_behavior": "v1 没有触发",
  "failure_pattern": "触发不匹配",
  "root_cause": "description 没覆盖创作者式稀疏表达",
  "suggested_action": "扩展 description 并加入回归集",
  "should_regress": true
}

09人机一致率：让分歧变成可优化信号

当 Skill 评测接入人评 / 机评体系后，很多 badcase 不应笼统归为“做错了”，而要进一步区分边界型分歧、严重度型分歧和平台运行型问题。

Boundary
边界型分歧
人认为该处理，模型认为不该处理，或反过来。重点动作是收敛入口、补白名单 / 黑名单、提升触发边界清晰度。
Severity
严重度型分歧
模型知道该处理，但判断太松，把 1 / 0 看成 2。重点动作是定义硬失败条件、补低分触发器、要求正反证据。
Platform
平台运行型问题
规则本身也许没错，但样本抽样、字段映射、调度执行出了偏差。重点动作是修输入链路、FieldMap、调度幂等与看板可观测性。

10治理：发布、拆分、退休都要有证据

治理的目标不是多留文档，而是让 Skill 可发布、可回归、可退休。它解决的是修改有证据、回归有抓手、发布有门槛、退休有依据。

资产作用没有它会怎样
all_cases全量样例池，承载历史积累每轮都像从头开始
disagreement_cases聚焦人机分歧与难样本最值得修的问题会被淹没
reason_archive错因 taxonomy 与行动建议失败只会重复出现，不会沉淀规律
regression_set每轮必测的黄金样例集修一处坏一片，没人第一时间知道
rule_versions记录每轮改了什么、为什么改、预期风险是什么团队无法复盘为什么变成今天这样
benchmark_history记录不同版本、不同模型、不同样本批次的结果无法判断收益是在变好，还是只是在漂移
 
{
  "skill_version": "v0.6.2",
  "changed_parts": ["description", "examples"],
  "main_hypothesis": "improve implicit trigger recall",
  "expected_risk": ["slightly more over-trigger"],
  "evaluated_on": "2026-06-08",
  "decision": "iterate"
}
信号说明建议动作
with_skill 明显优于 without_skill有稳定净增益发布或继续保留
略有收益但成本偏高价值存在，但实现不经济精简、脚本化或拆分
两者长期持平模型能力可能已追平考虑退休或降级为参考资料
with_skill 更差Skill 在误导 Agent停用并重写
子任务行为差异越来越大Skill 边界过宽、正文过载拆分成多个更清晰的 Skills

11识别假优化：别被平均数骗了

很多团队最容易掉进的陷阱，不是没优化，而是优化方向错了。pass rate 提高并不等于 Skill 更有价值。

False Optimization
典型假优化包括：pass rate 提高但误触发更高，分数提高但 Token / 时延暴涨，旧 case 提升但新 case 退化，更像 grader 偏好而不是真正目标行为，或者 without_skill 已经能过，Skill 反而成了上下文税。

12Harness Engineering：稳定性来自反馈，而不是堆上下文

写好 Skill 只能解决“给 Agent 什么能力包”。当任务变长、跨多轮甚至跨会话时，系统稳定性很快会卡在 Skill 之外，需要外层 Harness 补齐可观测性、反馈回路、记忆分层和上下文治理。

System Principle
Context Engineering 负责让 Agent 看得更对，Harness Engineering 负责让系统偏了也能回来。
外层能力解决什么问题典型做法与 Skill 的关系
可观测性知道系统做到了哪里、错在哪里trace、日志、工具轨迹、状态快照、质量看板没有它，就无法准确评估 Skill 的真实作用
反馈回路系统偏了以后能被拉回来测试、Lint、Review Agent、趋势巡检、定时回归Skill 的价值要通过反馈回路维持
记忆分层避免长期规则、经验、临时状态混在一起规则层、经验层、状态层分离治理防止 Skill 规则与其他记忆互相污染
上下文治理避免历史大块内容长期占坑渐进披露、文件系统外存、Context Offloading、压缩分层让 Skill 真正做到低上下文成本运行

13记忆分层与 Context Offloading

长期规则、自动沉淀经验、当前会话状态必须分层治理。否则规则互相冲突，当前任务噪声挤压长期约束，低价值历史信息不断侵占上下文。

Rules
规则层
长期稳定约束、团队规范、安全边界。它应该少变、清晰、可审计。
Experience
经验层
长期有效、可复用的偏好与隐性知识。它需要持续回流，但不能污染任务状态。
State
状态层
当前会话的临时上下文、任务过程信息。任务结束后应被归档、压缩或丢弃。

Context Offloading 给 Skill 设计的启发是：原始大块内容经常只是中间燃料，不是长期记忆本体；优化重点不是某一轮少几个 Token，而是减少大块信息在 50 轮任务里的累计占用面积。

14组织协作与平台化落地

角色主要责任最容易失守的点
Skill 作者定义能力边界，编写入口、流程、模板和脚本策略只顾正文，不顾触发边界和评测接口
评测负责人建设样例库、grader、benchmark 与回归集只看 pass rate，不做分层诊断
工程支持脚本化校验、trace 接入、自动化运行、看板与平台能力只做基础设施，不反向参与规则设计
业务 / 使用方提供真实 case，定义“有用”的标准，确认实际收益只给抽象需求，不给可验证标准
 
需求提出
  -> 场景澄清
  -> 先跑一次真实任务
  -> 抽象 Skill 草稿
  -> 手动试跑
  -> 建 MVP eval
  -> 跑 with_skill / without_skill 对照
  -> 修 description / body / scripts
  -> 发布候选版本
  -> 加入回归集
  -> 定期重跑与退休判断

15从 0 到 1 的最小落地方案

如果团队还没有完整平台，不必等万事俱备。可以先跑一个最小可行闭环，让 Skill 进入真实迭代。

发布前最低要素说明
清晰的 description包含 When to Use / When Not to Use / 负向边界
轻量但可执行的 SKILL.md主流程、检查项、引用文件和脚本调用路径
真实示例与脚本化检查点至少 1 个真实示例，至少 1 个确定性校验点
12 条以上 eval case至少 2 条 negative case，并包含边界与扩展样例
对照结果与治理资产with_skill / without_skill 对照、badcase 归因、版本记录与回归集
  1. 统一 Skill 结构模板。 至少统一入口、正文结构、references、scripts、evals、governance 目录。
  2. 把 description 评审纳入正式流程。 最高杠杆往往不在正文，而在路由入口。
  3. 优先把确定性步骤脚本化。 从最容易出错、最常重复的环节开始。
  4. 建立可持续增长的 regression set。 每修一个高价值 badcase，就纳入黄金样例池。
  5. 记录 benchmark history。 没有时间序列，就无法识别假优化与退化。

Final Statement
Skill 的长期价值，不在于“我终于写出一个很强的 Prompt”，而在于它能否在真实系统里被稳定触发、带来可验证净增益，并随着评测、反馈和治理不断演进，而不是不断膨胀。
