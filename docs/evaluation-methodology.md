# Skill 评测方法论

本项目的评测方法来自一个核心判断：Skill 评测不是证明它“能跑”，而是验证它是否持续带来可解释的净增益。

## 核心定义

```text
lift = with_skill 通过率 - without_skill 通过率
net_value = 质量提升 - Token / 时间 / 维护成本
```

因此，评测至少要回答六个问题：

1. 该触发时，能否触发。
2. 不该触发时，能否保持克制。
3. 触发后，最终结果是否更好。
4. 这种提升是否稳定，而不是偶然跑对一次。
5. 换表达、换噪声、换新样本时，是否仍成立。
6. 这种提升是否值得它带来的 Token、时延和维护成本。

## 六层评测模型

| 层级 | 核心问题 | 典型证据 | 优先优化对象 |
| --- | --- | --- | --- |
| 触发层 | 该触发时是否触发，不该触发时是否沉默 | trace、negative case、trigger recall | `description`、负向边界 |
| 结果层 | 最终产物是否可用 | 输出结构、文件存在、报告完整性 | 主流程、模板、schema |
| 执行层 | 是否遵守 Skill 规则与约束 | 工具、命令、流程是否符合预期 | `SKILL.md`、checklist |
| 效率层 | 是否更省、更稳、更少绕路 | token、耗时、重试次数 | 渐进披露、脚本化 |
| 泛化层 | 换说法、换场景、换噪声是否仍成立 | 边界样例、扩展样例、历史样例 | 样例分布、反例、边界 |
| 治理层 | 是否便于回归、定位、维护与退休 | 版本记录、失败归因、回归集 | governance assets |

当前代码实现了前五层的自动评分，并通过 `governance/` 目录承载第六层治理资产。

## 样例库设计

起步样例库不追求数量，而追求结构完整。本项目当前配比：

| 类型 | 数量 | 目的 |
| --- | ---: | --- |
| 核心场景 | 6 | 覆盖标准视频创意包生成 |
| 边界场景 | 2 | 覆盖稀疏输入、复杂输入 |
| 扩展场景 | 2 | 覆盖标题池、封面文案等局部请求 |
| 负向场景 | 2 | 捕捉新闻稿、项目计划等过度触发 |

每条样例都显式声明：

- `should_trigger`：是否应该触发 Skill。
- `case_type`：核心、边界、扩展或负向。
- `risk_tag`：该样例要保护的风险点。
- `brief`：可结构化消费的产品信息。

## 评分规则

每个维度采用 0 / 1 / 2 分制，避免单一平均分掩盖关键问题。

| 维度 | 通过证据 |
| --- | --- |
| 触发 | `actual_triggered == should_trigger` |
| 产出 | 标题 8-12 个、封面至少 6 个、包含 Demo、大纲和口播 |
| 风格 | 命中钩子词、包含旧新对比、封面行长合格 |
| 效率 | Token 相对基线变化处于可接受区间 |
| 泛化 | 边界和扩展样例的触发与产出同时成立 |

失败会被归类为：

- 触发不匹配
- 产出不完整
- 风格较弱
- 效率成本偏高
- 泛化风险

这些失败模式会进入报告，用于指导下一轮修改。

## 从失败样例到迭代动作

失败样例不是问题清单，而是优化驱动器。每个失败样例至少记录：

- `case_id`
- `run_config`
- `human_expected`
- `observed_behavior`
- `failure_pattern`
- `root_cause`
- `suggested_action`
- `should_regress`

默认迭代优先级：

1. 先改 `description`，解决路由入口问题。
2. 再改 `SKILL.md` 的关键规则。
3. 再补示例和 references。
4. 最后把可确定的检查下沉到 scripts。

## 本项目的迭代结论

v1 的问题主要集中在：

- 稀疏输入漏触发。
- 只要封面文案的局部请求漏触发。
- 新闻稿任务误触发。
- 风格规则不够硬，导致输出偏通用产品介绍。

v2 的修正动作：

- 在 `description` 中增加正向和负向触发边界。
- 在 `SKILL.md` 中明确产出契约。
- 增加标题数量、封面行长、Demo 优先的检查项。
- 增加 `validate_package.py` 做确定性结构校验。

最终结果：

| 指标 | v1 | v2 |
| --- | ---: | ---: |
| 通过率 | 75.0% | 100.0% |
| 触发召回率 | 80.0% | 100.0% |
| 过度触发率 | 50.0% | 0.0% |
| 平均 Token | 1142 | 935 |
