TL;DR
本轮评测用同一批 12 条样例对比 without_skill、with_skill_v1、with_skill_v2 三种状态。 结果显示：Skill v2 将通过率提升到 100.0%，触发召回达到 100.0%，过度触发降到 0.0%， 同时相对 v1 平均 Token 下降 18.2%。结论：当前版本可以作为 MVP 发布，并继续沉淀真实回归样例。

01评测摘要

评测对象为 ai-video-creator-style。这个 Skill 的任务是把 AI 产品 brief 转成适合 B 站技术区的视频标题、 封面文案、Demo 方案、三段式大纲和口播稿。

本次报告关注的不是单次文案是否漂亮，而是 Skill 是否具备工程资产所需要的稳定性： 该触发时触发、不该触发时克制、触发后带来可量化净增益，并且失败样例能驱动下一轮迭代。

Interpretation
v1 证明 Skill 能显著提升产出完整度，但也暴露出 description 过宽、边界较弱和成本偏高。 v2 的价值在于把经验改动变成可回归的工程动作，而不是继续凭感觉调提示词。

04v1 到 v2 的变化

  • Description 收紧。 增加负向边界：不处理新闻稿、项目计划、issue 分诊或通用非视频写作。
  • 产出契约硬化。 SKILL.md 明确要求标题、封面文案、Demo 方案、大纲、口播稿和自检。
  • 检查项前置。 标题数量、封面行长、Demo 优先开场、旧方式和新方式对比成为必检项。
  • 脚本化验证。 新增确定性校验脚本，用于检查创意包结构和文案行长。

06治理资产与下一步动作

本项目已经把一次评测结果沉淀成可维护资产，而不只是一次演示输出。

  • evals/cases.json：12 条种子评测样例。
  • runs/iteration-001/benchmark.json：基准汇总。
  • runs/iteration-001/*/trace.jsonl：路由和参考资料加载轨迹。
  • runs/iteration-001/*/provider.json：Provider、模型和 usage 信息。
  • providers/：统一 mock、OpenAI、Anthropic 的模型运行适配层。
  • governance/versions.json：版本假设、风险、指标和决策。
  • governance/reason_archive.json：失败样例根因归档。
  • governance/regression_set.json：必须保留的回归样例。
  1. 补充真实样例。 从历史项目中加入 3-5 条真实产品信息作为回归样例。
  2. 稳定真实模型回归。 将 OpenAI / Anthropic provider 接入稳定回归环境，补齐密钥管理和费用上限。
  3. 引入 LLM Judge。 用作风格评价补充，但保留确定性检查作为主验收。
  4. 持续记录 v3。 在 governance/versions.json 中记录下一轮变化。
  5. 保留负向样例。 持续防止 Skill 对新闻稿、项目计划等相邻任务过度触发。
