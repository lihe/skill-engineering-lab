# Demo Design

Use this reference when designing the video hook, opening result, and product demo.

## Contents

- Core Principle

## Core Principle

The demo is the proof.

Do not start by asking "what features does the product have?" Start by asking:

- What can viewers see?
- What surprising result can happen in 10 seconds?
- What task proves the core product value?
- What old workflow does this demo collapse?

## Demo Quality Checklist

A strong demo should satisfy all four:

1. One-sentence explainable.
2. Has contrast, humor, or a visible result.
3. Naturally reveals the product's core capability.
4. Can be shown on screen without too much explanation.

If not, redesign the demo.

## Opening Result Types

### Type A: Runnable Thing

Best for AI coding, app builders, agents, no-code tools.

Examples:

- A playable game.
- A generated webpage.
- A mini app.
- A report dashboard.
- A deployed tool.

Opening structure:

1. Show the finished thing.
2. Say it was made through natural language.
3. Reveal the product.
4. Promise to show the process.

Use when:

- The product can create or modify software.
- The viewer can understand the result immediately.

### Type B: Visual Shock

Best for AI video, 3D, image, design, multimodal models.

Examples:

- Complex motion video.
- High-detail 3D asset.
- Stable character movement.
- Before/after image or model output.

Opening structure:

1. Show the strongest visual output.
2. Say what makes it difficult.
3. Clarify it is AI-generated.
4. Name the model/tool.

Use when:

- The output itself carries the emotion.
- Viewers need to inspect quality.

### Type C: Worker Scene

Best for productivity tools, AI assistants, CLI tools, voice tools, office workflows.

Examples:

- Excel game or report.
- Meeting summary.
- PPT generation.
- File translation.
- Subtitle extraction.

Opening structure:

1. Show a familiar work pain.
2. Flip it into an AI-powered result.
3. Explain the old workflow briefly.
4. Show the AI doing it faster.

Use when:

- The product helps ordinary work.
- The viewer should think "I need this."

### Type D: Bottleneck Visualization

Best for cloud, compute, storage, training, infrastructure.

Examples:

- GPU waiting for data.
- CPU running a model faster than expected.
- One AI assistant running 24/7 on cloud.
- Training moving from single GPU to cluster.

Opening structure:

1. Name the hidden bottleneck.
2. Make it visible through a metaphor or test.
3. Show the new architecture/product.
4. Translate technical improvement into speed/cost/stability.

Use when:

- The product is abstract.
- The viewer cannot directly see the backend value.

### Type E: Voice / Hardware Moment

Best for hardware, recording devices, voice models, local AI.

Examples:

- A cloned voice speaking another language.
- A card-sized device becoming an assistant.
- A meeting turned into a report.
- A noisy song transcribed accurately.

Opening structure:

1. Show the sound/object.
2. Show one practical workflow.
3. Show AI post-processing.
4. Explain long-term value.

Use when:

- The product is tactile or audio-first.
- The key value is "always present" or "second brain".

## Product-Type Demo Guidance

### AI Coding / Agent Tool

Best demo ideas:

- Build a small game.
- Build an office prank/productivity hybrid.
- Convert a file into a polished output.
- Let AI read a project and generate a report.
- Ask AI to modify an existing output.

Required beats:

1. Natural-language prompt.
2. AI planning or generating.
3. Result shown.
4. Second-round modification.
5. Value summary: from prompt to runnable result.

Avoid:

- Only showing code.
- Only talking about model scores.
- A demo that requires viewers to understand programming details.

### AI Video / 3D / Creative Model

Best demo ideas:

- High-motion scene.
- Multi-subject movement.
- Texture or material close-up.
- Camera movement test.
- Workflow-ready asset export.

Required beats:

1. Show output first.
2. Explain why this scene is hard.
3. Compare with common AI failure mode.
4. Show controllability or iteration.
5. Value summary: not just beautiful, usable.

Avoid:

- Dark or overly cropped visuals.
- Outputs that require long explanation.
- Generic "AI can generate video now" claims.

### Cloud / Infrastructure

Best demo ideas:

- Cost comparison.
- Speed benchmark.
- Deployment walk-through.
- Customer case with before/after.
- Architecture bottleneck animation.

Required beats:

1. Turn invisible infrastructure into a visible problem.
2. Show test or case.
3. Translate parameters into outcomes.
4. Connect to real production or creator workflow.

Avoid:

- Pure parameter reading.
- Too many product names in a row.
- Architecture without a viewer-facing pain.

### Voice / Hardware

Best demo ideas:

- Voice clone.
- Meeting summary.
- Noisy transcription.
- Cross-file Q&A.
- Audio-to-podcast transformation.

Required beats:

1. Show the sound/object.
2. Show one practical workflow.
3. Show AI post-processing.
4. Explain long-term value.

Avoid:

- Only showing hardware appearance.
- Listing every feature.
- Ignoring the "after recording" AI value.

## Demo Before Script

Before writing the script, produce a demo plan:

```markdown
## Demo Plan
- Opening result:
- Viewer surprise:
- Product capability proven:
- On-screen steps:
- Second-round iteration:
- Old way vs new way:
```

If the demo plan is weak, do not write the final script yet.
