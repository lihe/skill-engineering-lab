# Outline Patterns

Use this reference when creating outlines for AI product videos.

## Contents

- Default Structure
- Part 1: Product Introduction
- Part 2: Features + Real Test
- Part 3: Value Summary + CTA
- Structure Variants By Product Type
- Outline Output Format

## Default Structure

Use a three-part structure by default:

1. 产品引出
2. 亮点 + 实测展示
3. 价值总结 + 行动号召

The viewer journey is:

1. See a surprising result.
2. Understand what made it possible.
3. Believe the product can change their workflow.
4. Feel invited to try it.

## Part 1: Product Introduction

Goal: create entry, not product background.

This section should usually end with a brief product intro. The opening still belongs to the result, pain, or contrast; the product intro is the landing point of part 1, not the first sentence.

This section should answer:

- What did we make or test?
- Why is it surprising?
- What old assumption does it break?
- What product/tool made it possible?

Common openings:

- Demo first: show a runnable app, generated video, 3D model, voice clone, report, or workflow result.
- Pain question: ask a viewer-facing question.
- Worker scene: use Excel, KPI, PPT, meeting, boss, report, project, deadline.
- Old vs new: explain what used to be difficult and what is now easy.

Recommended order:

1. Show the result.
2. Point out the contrast.
3. Clarify it is not a template/manual edit/fake demo if relevant.
4. Briefly introduce the product at the end of part 1.
5. Set up the test question.

Product intro at the end:

- Use 1-2 content points only.
- Include product name, one-sentence positioning, and the specific capability used in this video.
- Avoid company history, full feature lists, parameter stacking, or generic brand praise.
- In a client-facing outline, make this the final content point of the first section.

Common lines:

- 这是我刚刚用 [产品] 做出来的。
- 简单说，[产品] 是一个 [定位]，这次我主要用它完成 [本期任务]。
- 重点是，我一行代码都没写。
- 它不是 [旧理解]，而是 [新能力]。
- 那它到底有多猛？我们接着看。

Length target:

- 180-250 effective spoken Chinese characters.
- Viewer should understand the hook within 10-20 seconds.

Client-facing outline format:

- Only include content points.
- Do not include labels such as "画面/demo", "口播重点", or "目标".
- Keep the section easy for the client to understand at a glance.

## Part 2: Features + Real Test

Goal: prove the product through tasks, not feature lists.

This section should answer:

- How do you use it?
- What task did you run?
- What happened during the process?
- Which product abilities appeared naturally?
- Why is this better than the old way?

Recommended order:

1. Briefly explain product positioning.
2. Show setup or first step.
3. Run one visible task.
4. Explain 2-3 key abilities while the task progresses.
5. Show iteration or comparison if possible.
6. Summarize the main capability.

Rules:

- Explain features through task flow.
- Do not list more than 4 major capabilities.
- Every capability should connect to a demo, use case, or viewer pain.
- If a technical parameter appears, translate it into an outcome.
- Use comparison to make invisible infrastructure visible.

Common lines:

- 接下来我们直接实测一下。
- 首先打开...
- 不用选模板，也不用复杂配置。
- 你可以看到，它已经开始...
- 这也是我觉得最关键的一点。
- 对比一下你就更清楚了。

Length target:

- 550-700 effective spoken Chinese characters.
- This is usually about 60% of the script.

## Part 3: Value Summary + CTA

Goal: move from product feature to workflow change.

This section should answer:

- What is the product's real value?
- What does it change for creators, workers, developers, or teams?
- Why should the viewer try it now?

Recommended order:

1. Summarize the product in one strong sentence.
2. Use "not X, but Y" to raise the value.
3. Mention activity, download, trial, offer, or registration only after value is clear.
4. End with a human-level insight, not a slogan.

This section should include a "一句升华": one short value-elevation sentence that turns the product from a feature into a workflow or creation-style change.

Do not put publishing hashtags, topic tags, platform notes, or operational reminders inside the outline. Put those in a separate publishing notes section.

Common lines:

- 到这一步，其实可以做一个很清晰的判断了。
- 它不只是 [表面功能]，而是 [真正价值]。
- 如果你也对 [场景] 感兴趣，真的可以去试一下。
- 未来的 [领域]，可能不再是先 [旧门槛]，而是先有一个想法...

Length target:

- 150-220 effective spoken Chinese characters.
- CTA should usually stay under 80 characters.

## Structure Variants By Product Type

### AI Coding / Agent / App Builder

Use when the product can build, modify, automate, or execute.

Outline:

1. Result hook: show a game, webpage, report, app, or workflow outcome.
2. Product intro: AI can move from instruction to execution.
3. Test flow: input natural language, watch planning/generation/deployment.
4. Iteration: ask it to modify or add features.
5. Value: ordinary users can turn ideas into runnable outputs.
6. CTA: try building one thing yourself.

Must include:

- A visible output.
- A natural-language prompt.
- A second-round modification.

### AI Video / 3D / Image / Creative Model

Use when the product's value is visual quality, movement, texture, speed, or controllability.

Outline:

1. Result hook: show the strongest visual demo first.
2. Contrast: clarify why this was hard before.
3. Product intro: name the model/tool.
4. Feature tests: motion, consistency, texture, speed, control, workflow readiness.
5. Comparison: old model, old workflow, or common failure mode.
6. Value: from "can generate" to "can use".
7. CTA: invite viewers to test with their own prompt/material.

Must include:

- Visual proof.
- One high-difficulty scene.
- One sentence about workflow value.

### Cloud / Compute / Storage / Training Infrastructure

Use when the product is abstract or backend-heavy.

Outline:

1. Pain hook: expensive, slow, blocked, unstable, hard to scale.
2. Visible metaphor: GPU waiting for data, CPU running model, AI assistant online 24/7.
3. Product intro: show the infrastructure solution.
4. Real case or test: benchmark, deployment, customer case, workflow.
5. Parameter translation: turn technical specs into speed/cost/reliability.
6. Value: infrastructure is what lets AI actually run in production.
7. CTA: try, deploy, register, or contact.

Must include:

- One viewer-understandable bottleneck.
- One measurable or concrete improvement.
- One translation from architecture to real benefit.

### Voice AI / Hardware / Assistant Device

Use when product value is long-term use, portability, voice, recording, or memory.

Outline:

1. Object or voice hook: show the physical device or surprising voice demo.
2. Contrast: not just recording, not just TTS, not just a gadget.
3. Product intro: what it is.
4. Scene tests: meeting, class, finance, legal, travel, podcast, inspiration.
5. AI layer: summarize, translate, search, identify speakers, chat across files.
6. Value: second brain / always-present assistant.
7. CTA: start using it in daily work or study.

Must include:

- One everyday usage moment.
- One post-processing AI capability.
- One long-term memory or assistant metaphor.

### Developer Community / MCP / Contest

Use when the goal is to activate users or developers.

Outline:

1. Fun result hook: a small agent or app with emotional pull.
2. Community intro: models/tools/ecosystem available.
3. Build flow: select MCP/tools, configure, connect, test.
4. Showcase: agent solves a real or funny task.
5. Low-barrier value: beginners can make useful or playful tools.
6. CTA: register, build, join contest.

Must include:

- A simple user story.
- A build path that seems achievable.
- A playful example.

## Outline Output Format

When generating an outline, use this format:

```markdown
## 标题方向
[1-2 sentences]

## 第一部分｜产品引出
- 内容要点:
- [point]
- [point]

## 第二部分｜亮点 + 实测展示
- 内容要点:
- [point]
- [point]

## 第三部分｜价值总结 + 行动号召
- 内容要点:
- [point]
- [point]
- 一句升华: [one sentence]
```
